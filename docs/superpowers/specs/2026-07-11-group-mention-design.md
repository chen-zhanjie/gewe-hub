# 群聊 @ 输入交互与发送修复设计

## 目标

修复管理后台群聊中选择成员后实际 @ 不生效的问题，并将 @ 交互改为在输入框键入 `@` 后选择群成员。

## 已确认的根因

GeWe 的 `postText` 要求同时提供：

1. 可见正文中的 `@名称`；
2. `ats` 字段中的成员 wxid（多个成员用英文逗号分隔）。

当前前端仅维护成员 wxid 数组并提交为 `mentions`，没有把 `@名称` 写入文本；服务端因此发送了 `content: "正文"` 和 `ats: "wxid_xxx"`，导致真实 @ 不生效。

## 交互设计

### 触发与筛选

- 仅在群聊文本输入框中启用。
- 光标位于一个未被空白字符分隔的 `@` 之后时显示候选浮层。
- 用户继续输入 `@` 后的关键词，按群成员的群备注、显示名、昵称和 wxid 过滤。
- 只能新建状态为 `active` 的群成员提及；`left` 和 `removed` 不出现在可选择候选中。
- 初版浮层锚定输入框底部，不引入富文本编辑器或 caret 像素定位。

### 选择成员

选择候选成员时，在当前 `@查询词` 范围内替换为：

```text
@名称 
```

- 名称优先级：群备注，其次显示名、昵称、wxid。
- 自动添加的空格是该真实 @ 元数据的标记位。
- 光标定位到自动空格之后。

### 编辑与取消真实 @

- 草稿使用纯文本和 mention token 区间元数据，不使用 `contenteditable`。
- token 保存 wxid、名称、`@名称` 文本范围和自动空格的位置。
- 用户删除自动空格时，只删除对应 token 的元数据；`@名称` 可见文本保留。
- 用户编辑 `@名称` 内部任一字符时，移除对应 token 元数据，避免 wxid 与文本失配。
- 用户在 token 外插入或删除文字时，token 区间随最小编辑差异重定位。
- 发送时只取仍带自动空格标记的 token 生成 `mentions`；不通过反向解析 `@名称` 猜测 wxid。

## 发送与审计

- 前端发送同一份草稿文本与有效 wxid 数组。
- 服务端文本发送保持 `content = text`、`ats = mentions.join(',')`，确保两项一起到达 GeWe。
- 本地出站标准消息保存结构化的 outbound mentions，供工作台回显、事件与审计使用。
- 对“引用消息 + 真实 @”不静默降级：当前引用路径是 `postAppMsg`，未证实支持 `ats`。本次实现禁止在有引用时发送有效 mentions，并提示用户取消引用后再 @。

## 非目标

- 不实现 @全体成员。
- 不修改入站消息 mentions 的解析策略。
- 不引入 `contenteditable`、第三方富文本编辑器或复杂的 caret 像素定位。
- 不在输入框中实现服务端分页检索；初版使用已加载群成员过滤。

## 验收条件

1. 群聊键入 `@` 时出现 active 成员候选；私聊不会出现。
2. 选择成员后自动插入 `@名称 `，发送正文包含该文本且请求 `mentions` 包含对应 wxid。
3. 删除自动空格后，`@名称` 文本保留，但发送请求不再包含对应 wxid。
4. 编辑 `@名称` 本体后不保留陈旧的真实 mentions 元数据。
5. 服务端映射的 GeWe 请求同时含 `content` 的 `@名称` 和 `ats`。
6. 本地出站消息的标准 payload 保存该成员的结构化 mention。
7. 引用状态下选择或发送真实 @ 会得到明确阻止提示，避免假成功。
