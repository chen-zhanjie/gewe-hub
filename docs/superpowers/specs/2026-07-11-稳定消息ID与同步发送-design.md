# 稳定消息 ID 与同步发送设计

**日期：** 2026-07-11
**状态：** 已确认，进入实施

## 1. 目标

本轮在既有三态发送策略基础上，统一 GeWeHub 的消息身份、发送执行和 Agent 可见协议：

1. 所有新入站、出站消息创建时生成稳定 GeWeHub 消息 ID；
2. 发送成功后只补充平台 ID，不替换 GeWeHub 消息 ID；
3. AI 只使用稳定消息 ID 完成引用和撤回；
4. `/api/send` 同时支持同步和异步执行，默认同步；
5. 同步发送通过进程内全局串行协调器主动唤醒，不等待 2 秒轮询；
6. 标准发送响应不暴露 `pending`、`held`、`sendRequestId` 等内部状态；
7. 插件只标准化入参和透传，由 GeWeHub 决定发送、暂存或等待人工确认；
8. 删除旧布尔 `send` 和旧 ID 兼容分支，直接破坏性升级。

## 2. 标准消息 ID

### 2.1 格式

```text
msg_<22 位 Base64URL>
```

生成算法：

```ts
`msg_${randomBytes(16).toString("base64url")}`
```

ID 在 Message 首次创建时生成，永久不变。

### 2.2 数据职责

`Message` 是消息身份和平台映射的唯一事实来源：

```prisma
model Message {
  messageId           String @unique
  platformMsgId       String?
  platformNewMsgId    String?
  platformCreateTime  String?
}
```

删除：

- `Message.rawMessageId`
- `SendRequest.resultMsgId`
- `SendRequest.resultNewMsgId`
- `SendRequest.resultCreateTime`

`SendRequest` 继续承担请求、策略、执行状态、错误和原始响应记录，不再承担消息平台身份。

### 2.3 去重与历史数据

- 去重继续依赖 `dedupeKey`，不依赖消息 ID 格式；
- 入站重放先按 `dedupeKey` 查找，存在则复用原 Message；
- 新入站消息首次持久化时生成稳定 ID；
- 历史消息不批量改写 `messageId`；
- 迁移把历史 `rawMessageId` 写入 `platformNewMsgId`；
- 迁移把发送结果三字段写入关联 Message 的平台字段；
- 新代码只做 `Message.messageId` 精确查询，不判断前缀、不接受平台 ID、不接受数据库行 ID。

## 3. 发送策略与执行模式

两个维度独立：

```ts
deliveryMode: "immediate" | "discard" | "confirm"
executionMode: "sync" | "async"
```

`deliveryMode`：

| 值 | 当前行为 |
|---|---|
| `immediate` | 进入可靠发送队列 |
| `discard` | 创建未发送 Message，不调用平台 |
| `confirm` | 创建待确认 Message，不调用平台 |

`discard` 和 `confirm` 当前都沿用 held 行为，可由管理端人工发送；后续再分化业务规则。

`executionMode`：

| 值 | 行为 |
|---|---|
| `sync` | 默认。等待该消息实际发送成功、失败或结果未知后返回 |
| `async` | 持久化并可靠入队后立即返回，不等待平台结果 |

`discard/confirm` 不发生平台发送，因此无论执行模式都在 Message 和 SendRequest 创建完成后立即成功返回。

## 4. 出站写入流程

### 4.1 共同前置流程

`POST /api/send` 在一个事务内完成：

1. 校验标准请求；
2. 解析会话和引用消息；
3. 生成稳定 `messageId`；
4. 创建完整的本地 Message，`isSent=false`；
5. 创建 SendRequest 并关联 Message；
6. `immediate` 创建 send outbox；
7. `discard/confirm` 保持 held，不创建 outbox。

这样任何返回、失败、超时或人工重试都指向同一个稳定 Message。

### 4.2 全局串行协调器

当前实例只有一个发送 worker，继续保持所有会话全局串行，不做按会话并行，也不引入数据库分布式锁。

协调器职责：

- 维护一个进程内单飞 drain Promise；
- 新 send outbox 创建后主动 `wake()`；
- drain 按数据库 FIFO 连续领取并执行 send task，队列为空才结束；
- 同一时刻最多执行一个实际 GeWe 发送；
- 空闲时保留 2 秒定时 tick 作为进程重启和漏唤醒恢复机制；
- tick 与 wake 共用同一个单飞入口，不可重入；
- HTTP 连接中断不取消已持久化任务。

outbox 仍是可靠性边界，进程内协调器只负责低延迟和同步等待，不替代持久化队列。

### 4.3 同步等待

协调器按 `sendRequestId` 注册一次性 waiter：

- 成功：Message 原地更新平台三字段、`isSent=true`、实际 `sentAt` 和最终资源 URL，然后 resolve；
- 明确失败：Message 保留且 `isSent=false`，SendRequest 标记 failed，然后 reject `SEND_FAILED`；
- 平台超时或结果未知：Message 保留，SendRequest 标记 unknown，然后 reject `SEND_RESULT_UNKNOWN`；
- HTTP 同步等待上限 60 秒；等待超时不取消队列任务，不自动重试，并按结果未知返回。

同步请求通过内存 waiter 接收本次执行结果，不轮询数据库，不等待 2 秒 tick。

## 5. 标准响应

成功响应：

```ts
{
  success: true;
  messageId: string;
  url?: string;
  accepted?: true;
}
```

- 同步成功：返回 `success`、`messageId`，资源准备完成时返回统一 `url`；
- 异步入队：额外返回 `accepted:true`；若 URL 在入队时已经确定可返回，否则省略；
- `discard/confirm`：返回 `success`、`messageId`，不暴露 held 状态。

失败响应使用非 2xx：

```ts
{
  success: false;
  messageId: string;
  error: {
    code: "SEND_FAILED" | "SEND_RESULT_UNKNOWN";
    message: string;
  };
}
```

Agent 标准响应不再包含：

- `id`
- `status`
- `deliveryMode`
- `pending` / `held`
- `sendRequestId`
- `htmlPageId`
- `htmlHosted`
- `htmlPublicUrl`

所有可访问资源统一使用 `url`：HTML 页面、图片、文件、语音、视频和链接均遵循同一字段。

## 6. 引用与撤回

### 6.1 引用

发送请求继续使用：

```json
{ "replyToMessageId": "msg_xxx" }
```

服务端只按当前会话内 `Message.messageId` 精确查询，再读取：

- `platformMsgId`
- `platformNewMsgId`
- `platformCreateTime`

定位不能依赖 webhook raw payload、旧 `rawMessageId` 或去前缀逻辑。为生成 GeWe 引用消息体所需的原始 XML/正文，可继续读取关联 webhook payload，但它不是身份来源。

### 6.2 撤回

新增：

```http
POST /api/messages/:messageId/revoke
```

验证：

1. Message 存在；
2. `isSelf=true`；
3. `isSent=true`；
4. 平台三字段完整；
5. 调用 GeWe 撤回；
6. 原地更新 `status=revoked` 和 `revokedAt`。

删除旧 `/api/send/:sendRequestId/revoke`。插件新增 `gewehub_revoke_message(messageId)`，AI 不再接触 SendRequest ID 或 GeWe 原始 ID。

## 7. Markdown 上下文标准

```md
[上下文]
消息ID: msg_xxx
会话ID: cvs_xxx
会话类型: group
会话名称: 项目交流群
会话备注: 重点客户群
发送者ID: wxid_xxx
发送者名称: 张三
发送者备注: 张总
时间: 2026-07-11 10:30

[正文]
……
```

规则：

- 名称与备注分行；
- 缺失字段省略对应行；
- 私聊和群聊使用同一结构；
- 消息 ID 是唯一引用/撤回参数；
- 会话 ID 用于发送工具；
- 发送者 ID 用于 mentions；
- 不输出平台原始 ID、数据库行 ID 或 SendRequest ID；
- 嵌套引用与合并转发中的 `sourceMessageId` 也必须是稳定消息 ID。

## 8. 插件职责

插件统一通过一个 outbound 标准化和发送入口：

- 接受文本或 JSON；
- 普通 final 文本默认转成 `type=text`；
- JSON final 只要符合标准消息结构即可发送，不要求 final 必须 JSON；
- 透传 `deliveryMode` 和 `executionMode`；
- 默认 `deliveryMode=immediate`、`executionMode=sync`；
- 删除 boolean `send` 兼容；
- 只解析标准响应 `success/messageId/url/accepted`；
- 新增稳定 ID 撤回工具；
- 平台提示词明确引用和撤回只使用上下文中的 `消息ID`。

## 9. 管理端

保持已实现的未发送视觉和人工发送能力：

- `discard` 显示“未发送”；
- `confirm` 显示“待确认”；
- 低透明度、灰色、虚线边框；
- 操作区提供发送按钮；
- 发送期间禁用重复操作；
- 成功后按实际 `sentAt` 重排。

同步发送失败或结果未知的 Message 也保留，管理端可复用同一 SendRequest 重新派发，成功后仍更新原 Message 和稳定 ID。

## 10. 非目标

- 不做多实例分布式发送锁；
- 不改成按会话并行；
- 不为旧 boolean `send`、raw ID、SendRequest 撤回保留兼容解析；
- 不批量重写历史 Message ID；
- 不引入复杂自适应轮询；
- 不在本轮继续做无关服务端架构优化。
