## 1. 背景

GeWeHub 当前 Web 已具备聊天工作台、微信账号与通讯录、应用管理、推送日志、发送记录、HTML 页面、运行观测和接入设置等能力，但页面主要按桌面端设计：固定侧栏、工作台多栏布局、表格、右键菜单、Hover 操作和右侧详情 Sheet 在手机上难以操作。

本期目标不是创建一套新的产品功能，而是在**不改变现有业务能力和接口语义**的前提下，提供适合手机触控、单手操作和窄屏阅读的移动端展示层。

## 2. 产品目标

1. 用户通过主域名后的 `/mobile` 路径进入移动端。
2. 会话和聊天体验整体参考微信，降低移动端学习成本。
3. 通讯录成为移动端一级入口，展示当前微信账号已有的联系人和群聊数据。
4. 桌面端 Hover、右键菜单改为手机长按和底部操作面板。
5. 长列表、长表单、JSON 和多分区详情使用独立页面；短操作使用底部抽屉；危险操作使用确认弹窗。
6. 最大化复用现有 API、查询、业务控制器、数据模型、消息渲染与状态逻辑。
7. 保持现有桌面端路径和行为不变。

## 3. 非目标

本期不新增当前 Web 或服务端没有形成完整闭环的业务能力，包括但不限于：

- 消息转发；
- 聊天记录搜索或聊天内搜索；
- 拍照入口；
- 联系人标签、常用联系人；
- 修改微信群名、邀请或移除群成员；
- 语音转文字；
- 删除联系人；
- 删除服务端消息；
- 联系人详情中的新增编辑能力；
- 保存媒体到相册；
- 地图应用跳转；
- 群成员管理；
- 将会话管理中尚无真实数据的投递统计占位作为移动端核心能力。

通讯录成为一级页面属于对现有联系人和群列表的移动端重组，不代表新增后端能力。

## 4. 技术与产品路线

### 4.1 推荐路线

移动端不新建独立构建项目，而是在现有 `web` 应用中建立独立移动端展示层：

```text
web/src/features/mobile/
├── layout/
├── routes/
├── conversations/
├── contacts/
├── admin/
└── ui/
```

路由统一挂载到 `/mobile/*`。这样可以直接复用当前登录态、API Client、TanStack Query 缓存、SSE 连接、消息数据映射、发送控制器和构建部署流程。

### 4.2 复用原则

**复用数据和行为，不强求复用桌面页面 DOM。**

优先复用：

- `web/src/lib/api.ts`
- `web/src/lib/workspace-data.ts`
- `web/src/features/workbench/queries.ts`
- `web/src/features/admin/queries.ts`
- `web/src/features/workbench/message-timeline.ts`
- `web/src/features/workbench/mention-draft.ts`
- `web/src/features/workbench/workbench-local-sends.ts`
- 工作台发送、撤回、人工发送、会话操作和群成员控制器
- `web/src/components/message/MessageNodeView.tsx`
- Avatar、StatusBadge、TimeText、JsonViewer、CopyButton 等基础组件的逻辑与视觉语义

移动端单独实现：

- MobileShell；
- 底部 Tab；
- 移动端顶部栏；
- 会话与通讯录列表项；
- 手机聊天页面布局；
- 长按操作面板；
- 底部抽屉；
- 移动端表单页和详情页容器；
- 表格到摘要卡片的转换。

## 5. 信息架构

### 5.1 底部导航

| Tab | 职责 | 数据范围 |
|---|---|---|
| 会话 | 最近会话与聊天 | 当前微信账号 |
| 通讯录 | 联系人、群聊、同步和发起聊天 | 当前微信账号 |
| 管理 | 现有管理模块入口 | 全平台 |
| 我的 | 当前管理员、连接状态、账号切换、接入设置、退出 | 全局 |

聊天、详情、表单等二级页面不显示底部 Tab，以保证有效空间和正确返回语义。

### 5.2 路由

```text
/mobile/login
/mobile
/mobile/conversations
/mobile/conversations/:conversationId
/mobile/conversations/:conversationId/manage
/mobile/conversations/:conversationId/members
/mobile/conversations/:conversationId/messages/:messageId
/mobile/contacts
/mobile/contacts/:wxid
/mobile/groups
/mobile/admin
/mobile/admin/apps
/mobile/admin/apps/new
/mobile/admin/apps/:appId
/mobile/admin/apps/:appId/bindings
/mobile/admin/accounts
/mobile/admin/accounts/new
/mobile/admin/accounts/:accountId
/mobile/admin/deliveries
/mobile/admin/deliveries/:deliveryId
/mobile/admin/send-requests
/mobile/admin/send-requests/:requestId
/mobile/admin/html-pages
/mobile/admin/observability
/mobile/admin/observability/tasks/:taskId
/mobile/me
/mobile/settings
```

规则：

- `/mobile` 登录后重定向至 `/mobile/conversations`。
- 未登录访问任意受保护移动端路由时跳转 `/mobile/login`。
- 桌面端 `/login`、`/workbench` 等路径保持不变。
- 公开 HTML 页面 `/h/:token` 保持现有行为，不套移动端管理壳。
- 列表筛选继续使用 URL search 参数，便于刷新、返回和跨模块跳转。

## 6.4 消息长按操作

手机端不展示桌面 Hover 按钮。长按消息后，根据现有消息状态显示动作：

- 引用；
- 查看详情；
- 人工发送（held 消息）；
- 撤回（符合现有撤回规则的自己消息）；
- 重试（本地失败消息）；
- 删除本地失败消息。

不加入当前消息流没有统一实现的复制文本和转发。

撤回使用二次确认弹窗。

## 6.5 附件与复杂发送

点击输入区附件按钮打开底部面板，入口严格对应现有能力：

- 录制语音；
- 导入语音文件；
- 图片；
- 文件；
- 视频；
- 链接；
- HTML。

承载方式：

- 图片、文件、语音文件：系统文件选择后进入现有待发送附件队列；
- 录制语音：沿用现有录音开始、停止并发送、权限和错误状态；
- 视频：独立移动页面或全屏抽屉，包含视频文件、可选封面和发送；
- 链接：独立移动页面，包含 URL、解析、标题、描述、缩略图 URL/文件和发送；
- HTML：独立移动页面，包含 HTML 内容、HTML 文件、HTML 地址三种现有来源，以及标题、描述和缩略图。

## 7. 交互承载规范

### 7.1 独立页面

用于：

- 会话、聊天、通讯录长列表；
- 群成员；
- 会话管理；
- 消息详情；
- 视频、链接、HTML 等长表单；
- 管理列表、详情和长表单；
- JSON 调试内容。

必须支持浏览器返回、刷新和可定位 URL。

### 7.2 底部抽屉

用于需要保留当前上下文的短操作：

- 微信账号切换；
- 会话长按菜单；
- 消息长按菜单；
- 群成员长按菜单；
- 附件类型选择。

抽屉支持点击遮罩关闭、下滑关闭和底部安全区。

### 7.3 确认弹窗

用于：

- 撤回消息；
- 解绑应用；
- 重置 Token；
- 停用应用；
- 停用账号；
- 重投；
- 取消发送重试；
- 归档 HTML 页面；
- 重试失败任务。

需要输入名称/wxid 的强确认必须沿用现有规则。

## 8. 状态与异常

所有主要页面至少覆盖：

- 首次加载；
- 加载失败与重试；
- 空数据；
- 搜索无结果；
- 操作提交中；
- 操作失败；
- SSE 断开和重连；
- 账号离线或非 active；
- 登录失效；
- 本地消息发送中、成功、失败；
- 历史分页中、失败和无更多数据。

SSE 断开使用顶部非阻塞提示，不清空当前页面。

## 9. 视觉与可用性

- 设计基准宽度：375px、390px、430px；最低支持 320px。
- 主体参考微信的信息密度、顶部栏、会话列表和聊天布局，不复制商标与素材。
- 主要字号不低于 14px，辅助信息不低于 11px。
- 所有主要触控目标不小于 44×44 CSS px。
- 适配 `env(safe-area-inset-top)` 和 `env(safe-area-inset-bottom)`。
- 底部输入区、抽屉和 Tab 不被系统安全区遮挡。
- 长文本、wxid、ID 和 JSON 可复制并正确换行或横向滚动。
- 支持 `prefers-reduced-motion`。
- 避免仅依赖 Hover、右键和键盘快捷键。

## 10. 数据与权限

- 移动端继续使用现有管理员认证和 Cookie/Session 机制。
- 不增加新的用户角色和权限模型。
- 不直接暴露新增原始数据；消息详情中 raw/payload 的权限与现有控制台一致。
- 当前微信账号选择继续持久化，但移动端使用独立存储 key，避免影响桌面选择。
- 所有更新动作继续调用现有 API，保持软状态规则和审计行为。
