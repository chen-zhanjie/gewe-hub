# 统一消息发送与待发送消息设计

> **已被后续方案取代：** 本文保留为阶段性设计记录。现行协议以 `2026-07-11-稳定消息ID与同步发送-design.md` 和对应 implementation plan 为准，旧 boolean `send`、占位消息 ID 与旧响应字段均已删除。


## 1. 背景

Hermes 插件当前在 `adapter.py`、cron、standalone、HTML metadata 和工具发送路径中分别处理 final JSON、文本、`send` 开关和标准消息参数，导致规则重复且行为不一致。当前 gateway final 甚至会抑制普通文本，要求 AI 必须输出裸 JSON。

本次调整保持 GeWeHub `POST /api/send` 的标准消息接口定位，但为标准发送请求增加布尔字段 `send`：

- `send` 缺省或为 `true`：按现有流程创建发送请求并进入 outbox。
- `send: false`：服务端仍创建发送请求和本地聊天消息，但不创建 outbox、不调用 GeWe。
- 管理端正常展示未实际发送的消息，并提供人工发送入口。
- 插件不再本地拦截 `send:false`，而是统一标准化后传给 `/api/send`。

## 2. 核心原则

1. `/api/send` 仍只接收标准化消息，不接收未解析的 AI 原始协议。
2. 插件负责把字符串或 JSON 输出标准化为现有 `SendRequest`；GeWeHub 负责记录、调度、平台发送和状态管理。
3. `send:false` 是“保留/待发送（held）”，不是失败、撤回或草稿删除。
4. 未发送消息和实际发送后的消息是同一条数据库消息；人工发送成功后更新原消息，不生成重复消息。
5. 消息时间线使用实际 `sentAt` 排序：未发送时先使用记录时间，真正发送成功后更新为 GeWe 返回的实际发送时间，前后端重新排序。
6. 本次不顺带重构 `/api/send` 的 discriminated union、URL 字段别名等其他协议问题。

## 3. 标准协议

### 3.1 请求

在现有 `SendRequest` 顶层增加：

```ts
send: boolean // 可选，默认 true
```

示例：

```json
{
  "conversationId": "cvs_xxx",
  "type": "text",
  "text": "已处理完成",
  "send": false,
  "idempotencyKey": "final-thread-1"
}
```

`send` 是标准请求的执行开关，不放入 metadata。其他字段仍按当前消息类型校验。

### 3.2 响应

`SendResponse.status` 增加 `held`：

```json
{
  "id": "send_xxx",
  "status": "held",
  "messageId": "msg_held_send_xxx"
}
```

### 3.3 人工发送

新增：

```http
POST /api/send/:id/dispatch
```

约束：

- 仅 `held` 请求可进入调度。
- 原子地将状态更新为 `pending` 并创建唯一 outbox task。
- 重复点击或并发请求不能创建重复 outbox。
- 非 held 请求返回明确冲突错误。

## 4. 数据模型

### 4.1 SendRequest

`SendStatus` 增加：

```prisma
held
```

含义：请求和聊天消息已落库，但尚未提交到 GeWe。

### 4.2 Message

增加：

```prisma
isSent Boolean @default(true) @map("is_sent")
```

规则：

- 入站消息和历史已发送消息迁移后默认 `true`。
- `send:false` 创建的本地消息为 `false`。
- 人工发送成功后更新为 `true`。

同时在标准 `MessageEnvelope.metadata` 写入：

```json
{
  "outbound": {
    "sent": false,
    "sendRequestId": "send_xxx"
  }
}
```

数据库列是查询和 UI 判断的权威来源；payload metadata 用于详情、导出和标准消息自描述。

## 5. 服务端处理

### 5.1 标准发送编排

本次按范围约束不重构服务端整体 send 架构，保留现有 `SendController` 的标准发送编排，只抽取可复用的标准消息构建方法：

- `buildLocalContentFromSendRequest()`：把标准发送请求转换为本地 MessageNode；
- `buildLocalHubSendMessage()`：统一生成已发送或待发送的本地标准消息；
- `enqueueSendRequest()`：统一创建 send outbox。

Controller 在现有会话解析、幂等、引用、HTML 和 GeWe 映射流程后，根据 `send` 分流为 pending/outbox 或 held/local message。其他服务端结构优化留到独立任务。

### 5.2 held 本地消息

`send:false` 时：

1. 创建状态为 `held` 的 `SendRequest`。
2. 生成稳定占位消息 ID：`msg_held_<sendRequestId>`。
3. 创建 `source=hub_send`、`isSelf=true`、`isSent=false` 的 Message。
4. `sentAt` 暂用服务端记录时间。
5. 使用标准消息节点渲染文本、图片、文件、链接、HTML 等内容。
6. 更新会话最近消息和消息数，但不创建 outbox，不调用 GeWe。

### 5.3 人工发送和成功回写

人工 dispatch 后复用现有 outbox。发送成功时：

- 如果该 SendRequest 已有关联 Message：更新原记录；
- 把占位 `messageId/rawMessageId/dedupeKey` 替换为真实 GeWe 结果；
- 更新 `sentAt` 为实际发送时间；
- 更新 payload 中的 `messageId/sentAt/metadata.outbound.sent`；
- 设置 `isSent=true`；
- 不增加 messageCount；
- 更新会话最近消息时间；
- 前端刷新后按新时间重排。

普通 `send:true` 没有预建 Message，继续按现有逻辑在发送成功后创建消息。

## 6. 插件重构

新增 `plugins/hermes-agent/outbound.py`：

```python
@dataclass(frozen=True)
class OutboundDecision:
    payload: dict[str, Any]


def normalize_outbound(
    conversation_id: str,
    value: str | dict[str, Any],
    *,
    final: bool,
    metadata: dict[str, Any] | None = None,
) -> OutboundDecision:
    ...
```

final 规则：

| 输入 | 标准化结果 |
|---|---|
| 普通文本 | `type=text,text=原文,send=true` |
| 非法 JSON 字符串 | 按文本发送 |
| 合法 JSON 对象但无布尔 send | JSON 原文按文本发送 |
| JSON `send:false` | 标准消息 payload，保留 `send:false` 并调用服务端 |
| JSON `send:true` | 标准消息 payload，调用服务端 |
| `send:true/false` 不是布尔值 | JSON 原文按文本发送 |
| `send:true` 但标准 payload 非法 | 插件返回明确错误，不泄漏协议 JSON |

兼容 `{"send":false,"content":"..."}`：插件将 `content` 规范为 `type=text,text=...`。

所有实际发送最终调用：

```python
GeWeHubClient.send_message_payload(payload)
```

需要本地文件时，插件先转成 Base64，再调用同一入口。

## 7. 管理端

### 7.1 数据映射

`MessageItem` 增加：

```ts
isSent: boolean;
sendRequestStatus?: string;
```

会话消息接口 include `sendRequest`，用于展示 held/pending/failed 状态。

### 7.2 视觉

未发送消息：

- 整体透明度降低；
- 气泡使用虚线边框；
- meta 区显示“未发送”；
- 不显示撤回按钮；
- 在撤回按钮所在操作区显示“发送”按钮。

### 7.3 交互

点击“发送”：

1. 调用 `POST /api/send/:sendRequestId/dispatch`。
2. 按钮进入 loading/disabled。
3. 成功后刷新当前会话消息和会话列表。
4. outbox 完成后通过现有轮询/刷新获得真实发送时间。
5. 消息按新的 `sentAt` 自动移动到时间线正确位置。

## 8. 兼容性与幂等

- 旧调用方不传 `send`，行为完全等同当前实现。
- 现有 `idempotencyKey` 查询对 held 请求同样生效，重复 final 不创建重复记录。
- held dispatch 使用状态条件更新防并发重复发送。
- 已 held 的消息人工发送后，原 idempotency key 仍对应同一 SendRequest。
- 插件工具调用默认为 `send:true`。

## 9. 非目标

- 不重构 `/api/send` 为 discriminated union。
- 不统一 `mediaUrl/fileUrl`。
- 不移除 `requestId` 别名。
- 不改变引用、@、HTML 托管、媒体转换的现有能力。
- 不让服务端解析 Markdown 代码块或自然语言中的 JSON。

## 10. 三态发送策略升级（2026-07-11）

布尔 `send` 已升级为正式字段：

```ts
deliveryMode: "immediate" | "discard" | "confirm"
```

发送策略与执行状态分离：

- `deliveryMode=immediate`：直接创建 `pending` 请求并进入 outbox。
- `deliveryMode=discard`：当前创建 `held` 请求和 `isSent=false` 消息，不进入 outbox；管理端显示“未发送”。
- `deliveryMode=confirm`：当前创建 `held` 请求和 `isSent=false` 消息，不进入 outbox；管理端显示“待确认”。

`discard` 与 `confirm` 当前均可通过人工发送接口 dispatch；后续迭代可以在不改变历史意图的前提下分别升级策略。

服务端在 `SendRequest.deliveryMode` 独立保存原始策略；`status=held|pending|sent|failed|unknown` 继续表示执行状态。标准消息 `metadata.outbound.deliveryMode` 保留相同信息。

旧布尔 `send` 仅作为兼容输入：`true -> immediate`、`false -> confirm`。同时传入时 `deliveryMode` 优先。插件、工具 schema、平台提示词和新文档统一使用 `deliveryMode`。
