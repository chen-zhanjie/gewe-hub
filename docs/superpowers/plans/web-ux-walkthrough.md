# GeWeHub Web 用户旅程走查

> 阶段：前端标准化优化阶段 1
> 日期：2026-07-07
> 依据：`docs/前端规范-v1.md`、`docs/产品形态-v1.md`、`docs/标准消息结构-v1.md` 与当前 `web/src` 实现。
> 目的：先从使用者视角确认体验断点，再进入组件重建和页面拆分。本文不是验收通过声明，而是后续阶段 2-4 的任务输入。

## 当前实现证据摘要

| 观察项 | 当前证据 | 结论 |
|---|---|---|
| 代码组织 | `web/src/features/workbench/WorkbenchPage.tsx` 2582 行；`web/src/features/admin/AdminPages.tsx` 1524 行 | 工作台和管理页已经成为高风险上帝文件，阶段 2 必须拆分 |

| 工作台消息 | 已有虚拟滚动、日期分隔、同发送者 3 分钟分组、文本乐观发送、投递状态点、消息调试 Tab | 不能再按“完全没有”处理，应在现有能力上补缺 |

## 旅程走查

| 旅程 | 我想做什么 | 现在要几步 / 看到什么 | 困惑或多余 | 目标形态 |
|---|---|---|---|---|
| 1. 首次登录 → 找到某个客户会话 → 看懂最近聊了什么 | 管理员进入系统后快速定位某个客户，并判断最近上下文 | 登录后进入控制台；左侧工作台会话列表有账号按钮、搜索框、头像、名称、最近消息、时间和绑定状态 Badge；默认选中第一个会话；中间消息流展示日期分隔和消息分组 | 会话列表没有未读数、置顶分区、异常状态图标；账号按钮看起来像可切换但第一版实际只显示首个账号；最近消息没有发送者前缀；空态仍是普通文本 | 会话列表成为真正的工作台入口：未读红点/99+、置顶区、最近消息发送者前缀、账号切换真实可用或明确单账号、空态用 `EmptyState`；当前会话打开后定位到底部或未读分隔线 |
| 2. 收到一条新消息 → 我如何感知到 | 新消息到达时，不论当前在哪个页面，都能知道是否需要处理 | 管理壳监听 `/api/admin/events`，连接断开时顶栏显示重连横幅；若事件带 message/conversation，工作台 query 会 patch，当前会话可追加消息 | 用户不在当前会话底部时没有“↓ N 条新消息”；其他会话来消息主要依赖列表摘要更新，没有明显未读点；在管理页时只看到顶栏连接状态，看不到业务新消息提醒 | 当前会话且在底部时直接追加；当前会话但不在底部时显示新消息浮标；其他会话更新摘要与未读点；管理页可在导航/命令面板里暴露未读入口；断线恢复后静默补数据并清掉横幅 |
| 3. 给会话绑应用并配防抖 → 确认配置已生效 | 把一个会话交给某个 Hub 应用，设置过滤和防抖后确认当前生效值 | 在工作台右侧“绑定与投递”面板选择应用、投递过滤、防抖毫秒、最大等待毫秒，点击“保存绑定”；会话信息里能看到绑定应用和过滤器 | 备注和绑定混在一个表单里；保存后主要靠列表刷新，没有明确“已生效”确认；防抖默认值/应用覆盖/会话覆盖关系不明显；解绑有确认但没有显示具体应用名 | 右侧详情栏分清“当前生效配置”和“编辑配置”；保存后局部乐观更新并显示具体成功反馈；防抖显示“会话覆盖 / 应用默认 / 未设置”；解绑确认写明应用名和后果 |
| 4. 一条消息投递失败 → 定位原因 → 重试 → 确认成功 | 从聊天气泡发现投递失败，查原因，必要时重投，再确认状态恢复 | 气泡角落有投递状态点，点击可打开消息调试；调试弹窗有基本信息、标准 JSON、raw、投递记录；投递记录可一键跳推送日志并带 `status=failed`/`messageId` 预筛选；推送日志页有状态筛选、详情 Sheet、重投确认；推送日志会话名可直达工作台对应会话 | 表格没有重试历史 Tab；重投后状态刷新还停留在列表 reload，没有更细的局部状态流转 | 气泡失败点 → 调试弹窗投递 Tab → 一键跳推送日志并带 `status=failed`/messageId 筛选；推送日志默认优先暴露失败；详情使用 `DetailSheet` + `DescriptionList` + payload/retry history Tabs；重投后状态精确刷新 |
| 5. 给群成员改备注 → 在消息流里立刻看到备注生效 | 维护群成员平台备注，并让后续/当前消息显示更易懂 | 右侧“成员群”可搜索群成员，成员行有头像、状态、wxid、备注输入框和保存按钮；保存后刷新群成员列表 | 已选消息流里的发送者名不会显式跟随成员备注刷新；备注编辑是整行表单感，每个成员按钮文案包含长 wxid；没有内联 `✓/×` 快速编辑；非 active 成员只灰显，没有“显示全部/仅 active”控制 | 成员备注使用内联编辑，Enter/Esc 支持；保存成功后当前消息流同 wxid 的发送者展示立即更新；非 active 成员默认折叠或可切换显示；按钮和错误提示不挤占成员信息 |
| 6. 后台发一张图片 → 发送中/成功/失败三态体验 | 在工作台发送图片，立即看到发送状态，失败可重试 | 图片按钮打开文件选择；媒体发送会调用 `/api/send`，成功后刷新当前会话；文本发送已有本地乐观消息、发送中、失败、重试、删除 | 媒体发送没有文本同等的乐观气泡；图片上传时输入区是全局 `sending`，用户只能等刷新；失败只在输入区显示错误，不知道是哪张图失败；附件条要手动确认但没有每个附件的发送状态 | 媒体和文本共享发送状态机：提交即插入本地图片/文件/语音气泡，显示半透明和时钟；成功用服务端消息替换；失败在对应气泡显示红色感叹号、重试、删除；发送框立即可继续输入 |
| 7. 查一条历史消息的原始 payload 并复制给同事 | 从聊天记录定位一条消息，复制标准 JSON 或原始 payload | 右键消息可查看详情、复制文本、复制标准 JSON、查看原始 payload；详情弹窗有标准 JSON、raw、投递记录 Tab 和复制按钮；历史消息可点“加载更早消息” | “查看原始 payload”和“查看详情”都打开同一个弹窗，不能直达 raw Tab；JSON viewer 仍偏基础，不支持路径复制、搜索 key、折叠层级；加载历史没有搜索/跳转指定消息 | 右键菜单动作直达对应 Tab；`JsonViewer` 支持搜索、折叠、路径复制；调试弹窗支持上一条/下一条；从推送日志或发送记录可回跳工作台并高亮消息 |
| 8. 微信账号掉线 → 我多久能发现 → 在哪看详情 | 账号离线时快速发现影响范围和处理入口 | 顶栏固定显示“GeWe 连接 online”；账号页已增加“微信账号状态摘要”，离线账号置顶并点名离线对象；运行观测有账号在线 `online / total` 指标 | 顶栏 GeWe 状态目前看起来是静态展示，不一定来自 `/api/gewe/status` 或账号汇总；账号页还缺最近更新时间；运行观测只给离线数量，不说具体账号和影响哪些会话/应用 | 顶栏连接状态来自真实状态；账号页离线优先并显示最近更新时间；运行观测顶部健康摘要直说“X 个账号离线：账号名”；可跳到账号详情或受影响会话 |
| 9. 每个管理页第一眼要回答什么问题 | 进入管理页时马上知道是否有异常、该做什么 | 应用页显示新建表单和应用卡片；账号页已有状态摘要并离线优先；推送日志默认聚焦失败并有状态摘要；发送记录有状态筛选和分页；运行观测有健康摘要、指标卡和失败任务列表；接入设置有 key/回调/地址指标 | 页面第一眼仍偏“表单堆叠”：应用页新建表单压在最前，列表信息密度不够；观测页失败任务重试是直接按钮无确认/详情；发送记录还缺失败摘要；管理页未完全统一 DataTable、Skeleton、EntityCell、TimeText | 每页顶部先回答核心问题：应用页=应用是否可用和接入信息；账号页=哪些账号离线；推送日志=有没有失败投递；发送记录=失败/可撤回消息；观测页=系统健康摘要；接入设置=回调是否可用。表格统一迁移 DataTable，详情统一 Sheet |

## 后续阶段任务边界

### 阶段 2：基础层重建优先级

1. 先拆文件，降低修改风险：`AdminPages.tsx` 按 apps/accounts/settings/deliveries/send-requests/observability 拆；`WorkbenchPage.tsx` 拆出 ConversationList、MessageFlow、MessageComposer、DetailPanel、MessageDebugDialog。
2. 补共享组件：`DataTable`、`EntityCell`、`TimeText`、`DescriptionList`、`EmptyState`、`DetailSheet`、`Skeleton`。
3. 管理页表格先迁移推送日志、发送记录、运行观测失败任务，因为它们直接支撑故障定位链路。
4. URL 状态从手写 `window.history.pushState` 迁移为 TanStack Router typed search；至少覆盖 `status`、`page`、选中详情 ID。

### 阶段 3：工作台体验优先级

1. 保留已有日期分隔、3 分钟分组、文本乐观发送和投递状态点。
2. 补齐媒体乐观发送、新消息浮标、其他会话未读点、调试弹窗直达 Tab 与上一条/下一条。
3. 将备注编辑改为内联编辑，并把保存结果同步影响当前消息流显示。

### 阶段 4：管理页信息设计优先级

1. 推送日志：失败优先、详情 Sheet、重投历史、跨页跳转。
2. 发送记录：详情 Sheet、撤回后状态刷新、按会话/消息回跳。
3. 运行观测：健康摘要直说异常对象；失败任务详情和重试确认。
4. 应用/账号/接入设置：从表单堆叠改为“状态摘要 + 表格/详情 + 明确主动作”。
