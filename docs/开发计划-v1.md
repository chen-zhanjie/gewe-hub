# GeWeHub 开发计划 v1

> 状态：定稿（2026-07-06）
> 依据：`docs/产品形态-v1.md`、`docs/标准消息结构-v1.md`、`docs/技术选型-v1.md`。
> 本文档是交付给开发 agent 的执行计划。按里程碑（M0~M6）顺序执行，每个里程碑有明确交付物和验收标准。全部完成后系统达到可用状态。
> 原则：第一版整体不过度复杂；与三份定稿文档冲突时，以定稿文档为准。

## 零、全局约束（所有里程碑通用）

1. 技术栈严格按 `技术选型-v1.md`：NestJS + Fastify adapter + MySQL 8 + Redis + Prisma；React + Vite + TanStack + Tailwind + shadcn/ui；Python Hermes 插件；Docker Compose。
2. 所有微信 ID / appId / newMsgId / messageId 全链路字符串，禁止 number。
3. 同步类数据软状态：只 upsert + 状态标记 + 时间戳，禁止物理删除。
4. 消息标准结构严格按 `标准消息结构-v1.md` 的信封 + 递归 MessageNode，类型集中枚举，禁止字符串字面量散落。
5. 中文注释和中文文档优先；代码标识符用英文。
6. 每个里程碑完成时必须能通过该节"验收标准"，且 `docker compose up` 后系统整体可运行（M2 起）。
7. 测试要求：核心链路（去重、标准化、投递状态机、发送状态机）必须有单元/集成测试；解析器测试用 `references/gewe-raw-samples/` 真实样本作为 fixture。

## 二、数据模型（Prisma schema，M0 定稿）

所有表含 `id`(cuid)、`created_at`、`updated_at`。关键表与字段（可加字段，不可少）：

```text

  platform_remark, last_synced_at, discovered_at

  account_id, wxid, nickname, avatar_url, platform_remark,

groups             群
  account_id, wxid(@chatroom), name, avatar_url, owner_wxid,

conversations      会话 =（微信账号, 对端）
  account_id, peer_wxid, type(private|group), name, avatar_url,
  platform_remark, app_id(nullable, 绑定的 Hub 应用),

  account_id, conversation_id, webhook_event_id(nullable), send_request_id(nullable),

  attempts, last_error, delivered_at, acked_at
  @@index([app_id, status])

send_requests      发送请求
  app_id(nullable, 后台发送为 null), account_id, conversation_id,
  type(text|image|file...), request_payload(json),
  gewe_request(json), gewe_response(json),
  status(pending|sent|failed), error_message,
  result_msg_id, result_new_msg_id, result_create_time  # 撤回三件套，字符串
```

### M1：webhook 接收 + outbox 框架 + 管理员登录

**交付物：**

1. **auth 模块**：`POST /api/auth/login`（env 账密，bcrypt 校验，签发 httpOnly cookie session 或 JWT）、`POST /api/auth/logout`、`GET /api/auth/me`。全部 `/api/*`（除 login/health/webhook/媒体签名 URL）走鉴权守卫。
2. **webhook 接收**：`POST /webhook/gewe/:secret`。
   - secret 不匹配 → 404；匹配 → 原始 payload 立即写 `webhook_events`，**3 秒内返回 200**（GeWe 硬性要求），一切处理异步。
   - 无 `msgType`/非法报文（订阅握手、探针）→ `event_kind=unknown`，`process_status=skipped`。
   - 计算 dedupe_key=`appid:newMsgId`（伪 id/缺失时按 `账号+sha1(稳定JSON)` 降级，见消息结构文档 4.3-8），重复直接 skipped。
   - 写入成功后插入 `outbox_tasks(process_webhook)`。
3. **outbox worker 框架**：NestJS 独立 worker（同进程 interval 轮询即可，无需独立部署）：
   - `SELECT ... FOR UPDATE SKIP LOCKED` 或 lease 字段抢占；
   - 失败指数退避重试（默认 max_retry=5），超限置 dead；
   - 按 task_type 分发到 handler 注册表；
   - 提供 `GET /api/outbox/tasks`（筛选状态）+ `POST /api/outbox/tasks/:id/retry`。
4. **GeWe client 模块**：封装 GeWe HTTP 调用（token header、统一错误、429/超时重试）。实现：setCallback、检查在线、获取个人资料、获取通讯录列表、获取群成员列表、各下载接口、各发送接口的方法签名（本里程碑只需 setCallback + 在线检查连通即可，其余方法后续里程碑用到时补全实现）。
5. **接入设置 API**：`GET /api/gewe/status`（key 有效性、回调 URL 展示）、`POST /api/gewe/set-callback`（一键设置回调）。

**验收：** 用 curl 重放 `references/gewe-raw-samples` 中 10 个样本到 webhook → 全部入库、重放重复样本被去重、非法报文 skipped；登录接口正确拒绝错误密码；outbox 任务失败后能看到重试。

### M2：标准化管线 + 媒体 + 账号/联系人/会话同步

这是核心里程碑，工作量最大。

**交付物：**

1. **账号自动发现**：process_webhook handler 中，新 `appid/wxid` → upsert `wechat_accounts(source=auto)`，异步调 GeWe API 补昵称/头像/在线状态。手动录入/编辑 API：`GET/POST/PATCH /api/accounts`。
2. **会话维护**：按（account, 对端）upsert conversation（群聊对端=fromGroup，私聊对端=fromUser/toUser），更新 last_message_*。
3. **normalizer 模块**（拆策略类，禁止上帝类）：
   - `parsers/text.ts`、`image.ts`、`voice.ts`、`video.ts`、`file.ts`、`emoji.ts`、`link.ts`、`mini-program.ts`、`quote.ts`、`chat-record.ts`、`location.ts`、`card.ts`、`transfer.ts`、`red-packet.ts`、`system.ts`、`fallback.ts`；
   - 严格实现 `标准消息结构-v1.md` 第三、四节：递归 MessageNode、引用三级降级（本地回查 svrid 优先）、合并转发 datatype 映射、APP_MSG(74) 不生成消息且 FILE(6) 按 overwrite_newmsgid 合并、全部 10 条解析陷阱；
   - mentions：pushContent 判 isAtMe；`@昵称\u2005` 提取 + 群成员表匹配，失败 `resolved:false`；
   - renderedText 生成规则集中一处。
   - **每个 parser 用对应类型的真实样本写测试**（样本目录按类型已分好）。
4. **media 模块**：
   - GeWe 下载（串行队列 + 3~10s 间隔 + 同消息只下一次）→ 落盘 `FILE_STORAGE_DIR/{accountId}/{yyyymmdd}/{random}.ext` → 生成带 HMAC 签名 + 过期时间的公开 URL（`GET /files/:id?sig=&exp=`）；
   - 语音 Silk→MP3（用 silk 解码 + ffmpeg，容器内置依赖）；
   - 下载失败有限重试（默认 3 次退避）；耗尽后**降级投递**：消息照常入库，`media.status:"failed"`、`url:null`、renderedText 给"[文件: xxx] 下载失败"类描述——不丢消息不堵队列；条目级失败同规则；后台提供失败媒体的人工重试下载（成功后更新 status/url，不推事件）。
5. **联系人/群/群成员同步**：
   - 手动触发 + 定时（每 6h）全量同步：通讯录列表、群成员列表 → diff → upsert + 软状态标记（本次未出现 → left/removed + 时间戳）；
   - MOD_CONTACTS/DEL_CONTACTS 回调触发单点同步；
   - API：`GET /api/contacts`、`GET /api/groups`、`GET /api/groups/:id/members`、备注编辑 PATCH（平台级备注）。
6. **撤回**：REVOKE_MSG → 解析定位 newMsgId → 原消息 status=revoked + revokedAt；撤回事件入库；若原消息未投递则不再投 created（M3 联动）。
7. **消息查询 API**：`GET /api/conversations`（分页/筛选）、`GET /api/conversations/:id/messages`（游标分页，支持 before/after 某 messageId 翻页——这就是给 AI 的聊天记录查询接口）、`GET /api/messages/:id`（含 raw payload、标准 payload、投递记录）。

**验收：** 将样本目录全部 178 个样本按序重放 → 消息表生成正确类型的标准消息（TEXT/QUOTE/CHAT_RECORD 等逐类抽查 payload 结构符合 schema）；QUOTE 样本能本地回查到被引用消息；FILE 两阶段回调只生成一条消息；媒体样本生成可访问的签名 URL；语音消息产出 MP3；重放 parser 单测全绿。

### M3：应用 + 路由 + SSE 投递 + 发送

**交付物：**

1. **apps 模块**：`GET/POST/PATCH /api/apps`（name、token 生成/重置、owner_wxid、main_conversation、默认防抖、deliver_self_messages）；应用级账号备注 CRUD；`GET /api/apps/:id/conversations`（绑定列表）。
2. **会话绑定**：`POST /api/conversations/:id/bind`（app_id、delivery_filter、防抖覆盖）、`POST /api/conversations/:id/unbind`。一会话一应用，改绑必须先解绑（服务端强校验）。
3. **delivery 模块**：
   - 消息标准化完成后：会话已绑定应用且通过过滤（delivery_filter=at_only 时仅 isAtMe；isSelf 按 deliver_self_messages）→ 组装投递 payload（**此时注入该应用视角的备注**：应用级账号备注优先回落平台级；会话/成员备注平台级）→ 写 `deliveries` + outbox 任务；
   - **per-conversation 有序**：同会话上一条 delivery 未完成（delivered）不投下一条（Redis per-conversation 锁 + DB 顺序校验），跨会话并行；
   - **SSE 端点** `GET /api/apps/events`（Bearer token 鉴权）：连接后先补发未 acked 事件（支持 `Last-Event-ID`），后实时推送；心跳注释帧每 15s；单 app 单活跃连接（Redis 锁，新连接顶掉旧连接）；
   - 事件格式：`id: {event_id}\nevent: {event_type}\ndata: {"eventId","eventType","payload"}`，payload 即标准信封，metadata 内附会话防抖配置（debounceMs/maxWaitMs，会话覆盖 > 应用默认）；
   - **ACK**：`POST /api/apps/events/ack` body `{eventIds:[]}` → status=acked；
   - `message.revoked` 投递给已收到该消息的应用。
4. **send 模块**：
   - `POST /api/send`（app token 或后台 session 鉴权）body：`{conversationId, type, text?, mediaUrl?/fileUrl?, fileName?, mentions?}`；
   - 第一版支持：text（含 ats）、image、file、voice、video、link；
   - 写 send_requests → outbox send 任务 → GeWe client 调对应接口 → 记录响应与撤回三件套 → 本地生成标准消息（source=hub_send，不回投给发送方应用）；
   - `POST /api/send/:id/revoke`（用三件套调 GeWe 撤回）；
   - `GET /api/send-requests`（记录查询）。
5. **observability**：`GET /api/observability/summary`（webhook 24h 计数、失败任务数、SSE 连接状态、投递积压、账号在线状态）。

**验收：** 集成测试跑通完整闭环——重放样本 → 绑定会话到测试应用 → 用 curl/脚本连 SSE 收到 message.created（payload 含备注与防抖 metadata）→ ACK 后事件不再补发 → 断线重连带 Last-Event-ID 收到断线期间事件 → 同会话按 sent_at 有序到达；at_only 过滤生效；调 /api/send 发文本 → send_requests 有记录且本地消息生成、SSE 不回投。

### M4：Web 工作台

> **强制约束：全部页面必须遵守 `docs/前端规范-v1.md`**（设计 token、布局、组件、状态设计、交互细节、代码组织）。该文档第八节的三类检查（禁止项扫描 / 截图对照 / 交互走查）是本里程碑验收的一部分。

**页面与交付物（全部走 contracts 类型 + TanStack Query）：**

1. **登录页**：账密登录，错误提示。
2. **聊天工作台（主屏，三栏）**：
   - 左栏：账号切换下拉 + 会话列表（TanStack Virtual，头像/名称/最近消息/时间/未绑定或应用徽标，搜索过滤）；
   - 中栏：消息流（虚拟滚动、向上翻页加载历史、按类型渲染气泡：text/image(缩略图点开)/voice(audio 播放)/video/file(下载)/link 卡片/mini_program 卡片/chat_record 折叠展开条目/quote 引用块/system 灰条/unsupported 占位/revoked "[已撤回]"徽标）+ 底部发送框（文本 + 图片/文件上传发送）；
   - 右栏 Tabs：会话信息（备注编辑、绑定应用操作：选应用/过滤器/防抖参数/解绑）、成员列表（群，灰显非 active，成员备注编辑）、调试（选中消息的标准 JSON / raw payload / 投递记录，均可复制）；
   - 消息右键菜单：查看详情/复制文本/复制标准 JSON/查看原始 payload。
3. **应用管理**：列表 + 新建/编辑（token 显示与重置、复制 SSE 接入配置片段一键复制）、绑定会话列表、应用级账号备注。
4. **接入设置**：GeWe key 状态、回调 URL 展示 + 复制、一键设置回调按钮、账号列表（含手动录入/编辑、同步按钮）。
5. **推送日志**：deliveries 表格（状态 Badge、筛选、重投按钮）。
6. **发送记录**：send_requests 表格（状态、GeWe 响应查看、撤回按钮）。
7. **运行观测**：summary 卡片 + 失败任务列表（重试按钮）。
8. 消息实时更新：工作台通过一条内部 SSE（管理员会话鉴权）接收新消息/撤回，刷新对应会话（简单实现：收到事件后 invalidate 对应 query 即可，不做前端消息流合并）。

**验收：** 浏览器完成全流程人工验收脚本——登录 → 看到重放样本生成的会话与各类型消息渲染正确 → 右键看 JSON → 编辑备注 → 绑定应用配防抖 → 发送文本和图片成功且气泡出现 → 推送日志看到投递记录 → 撤回一条已发消息后气泡变"[已撤回]"。

### M5：Hermes Agent 官方插件

**位置**：`plugins/hermes-agent/`。**直接参照 `/Users/agent/project/hermes-agent-pro/plugins/gewecenter-platform/` 的结构与写法**（它是已在生产验证的同类插件），按 Hermes 插件规范（`ctx.register_platform()` + `BasePlatformAdapter`）实现，**保持精简，第一版目标 <1500 行**：

```text
plugins/hermes-agent/
  plugin.yaml          # requires_env: GEWEHUB_BASE_URL / GEWEHUB_APP_TOKEN
                       # optional_env: GEWEHUB_HOME_CONVERSATION_ID / GEWEHUB_DEBOUNCE_MS 等
  adapter.py           # GeWeHubAdapter(BasePlatformAdapter)：SSE 循环、防抖、send_message
  client.py            # GeWeHub HTTP/SSE client（events、ack、send、媒体下载）
  normalizer.py        # 标准信封 -> Hermes MessageEvent 字段映射
  dedupe.py            # BoundedTTLSet 短期去重（照抄 gewecenter-platform 模式）
  state.py             # Last-Event-ID 持久化（本地文件）
  README.md            # 安装、配置、诊断说明（中文）
```

**核心行为（多数逻辑照搬 gewecenter-platform 已验证实现）：**

1. SSE 长连接 `GET {base}/api/apps/events`，Bearer token，断线指数退避重连，携带 `Last-Event-ID`。
2. 每事件：去重 → per-conversation asyncio.Lock 保序 → 处理成功后 ACK（合并批时全批处理完统一 ACK）。
3. **防抖**：读取事件 metadata 里的 debounceMs/maxWaitMs（0=关闭）。仅纯文本（无媒体无引用）攒批：滚动窗口重置计时，达 maxWaitMs 强制冲刷；非文本事件到达先强制冲刷保序；合并用 `\n\n` 拼 text，metadata 记 `batch.message_ids/count`。
4. **MessageEvent 映射**：`text=renderedText 或 content.text`；`source`: chat_id=conversation.id、user_id=sender.wxid、chat_type=conversation.type；媒体经 client 下载到本地临时文件后填 `media_urls`；quote → `reply_to_*`；`is_at_me`/`mentions`/备注等透传 metadata（**插件不做 @ 过滤**，过滤在 Hub 侧 delivery_filter，与 gewecenter-platform 分工一致）。
5. **send_message**：Hermes 回复 → `POST {base}/api/send`（文本；带附件时先按类型发 image/file）。
6. `message.revoked` 事件：仅记日志（第一版不通知 AI）。
7. `GEWEHUB_HOME_CONVERSATION_ID` 支持 cron/通知投递（`cron_deliver_env_var` hook）。

**验收：** 本地起 GeWeHub（M3 完成态）+ Hermes，配置插件后：向绑定会话发文本 → Hermes 收到并回复 → 回复出现在微信（或 GeWeHub 发送记录成功）；连发 3 条短消息（防抖 2s）→ Hermes 只触发一轮且 text 为合并文本；杀掉插件进程重启 → 不丢消息不重复处理（Last-Event-ID + 去重生效）；发图片 → Hermes 收到本地媒体文件路径。
