# GeWeHub 标准消息结构 v1

> 状态：已定稿（2026-07-06 消息结构讨论）
> 依据：178 个生产回调样本（`references/gewe-raw-samples/`）+ GeWe 官方文档 + GeWeCenter 实现经验 + Hermes Agent 消费模型调研。
> 与 `产品形态-v1.md` 同为第一版开发边界文档。

## 一、设计原则

1. **递归 MessageNode**：引用、合并转发、多层嵌套全部复用同一个节点结构。不为复杂消息散落特例类型，AI 通过稳定嵌套格式处理。
2. **三层结构**：`raw`（原始 payload，仅平台内部）/ `normalized`（标准层，本文档定义）/ `rendered`（展示文本，内置于标准层的 `renderedText`）。
3. **永不 null 炸弹**：每一层解析失败都有稳定降级结构（`unsupported` + 描述文本），下游永远拿到合法 JSON。
4. **类型集中枚举**：所有类型在代码中集中定义（enum/常量），禁止字符串字面量散落（GeWeCenter 教训）。
5. **ID 一律字符串**：`newMsgId` 是 int64，JSON 传输有精度丢失风险，全链路按字符串处理。
6. **原始字段不出平台**：CDN 参数（aeskey/cdnurl/fileId）、原始 XML 不投递给下游，媒体一律换平台代理 URL。

## 二、顶层信封（投递给下游的完整结构）

```jsonc
{
  "schemaVersion": 1,
  "eventType": "message.created",       // message.created | message.revoked
  "messageId": "msg_9154866412345678",  // 平台消息 ID，源自 newMsgId（字符串）
  "status": "normal",                    // normal | revoked
  "isSelf": false,                       // 是否托管微信自己发送
  "isAtMe": true,                        // 群聊中是否 @ 了托管微信（源自 pushContent，可靠）

  "account": {                           // 接收微信（哪个微信收到的）
    "wxid": "wxid_bot",
    "name": "机器人昵称",
    "remark": "客服号1"                  // 应用级备注优先，回落平台级
  },
  "conversation": {
    "id": "cvs_xxx",                     // 平台会话 ID
    "type": "group",                     // private | group
    "wxid": "48315023241@chatroom",
    "name": "客户群 A",
    "remark": "VIP客户群"                // 平台级会话备注
  },
  "sender": {
    "wxid": "wxid_abc",
    "name": "陈可乐",                    // 群成员显示名 > 联系人昵称 > pushContent 提取 > wxid
    "remark": "张总-决策人",             // 会话内成员备注（平台级）
    "isOwner": false                     // 是否应用配置的主人
  },
  "mentions": [                          // 群聊被 @ 列表
    { "wxid": "wxid_bot", "name": "机器人", "isMe": true, "resolved": true },
    { "name": "云知序", "resolved": false }   // 昵称匹配失败的降级（v2 协议无 atuserlist）
  ],

  "content": { /* MessageNode，见第三节 */ },
  "quote":   { /* MessageNode | null，被引用消息快照 */ },

  "renderedText": "@机器人 看看这个",    // 统一展示摘要（会话列表/UI/AI 上下文共用）
  "sentAt": "2026-07-05T20:49:00+08:00"  // ISO8601，统一自顶层 createTime（毫秒）
}
```

约定：

- `isAtMe` 判定来源：`pushContent` 含"在群聊中@了你"或"@了所有人"——这是 v2 回调下唯一可靠信号。
- `mentions` 其余成员：从文本 `@昵称\u2005` 提取展示名，尽力匹配群成员表得 wxid（`resolved: true`）；匹配失败保留展示名（`resolved: false`）。GeWe v2 回调无 `MsgSource/atuserlist`，这是协议上限。
- `sender.remark` 等备注字段按投递目标应用组装：微信账号备注取应用级优先回落平台级；会话/成员备注只有平台级（见产品形态 4.2）。

## 三、MessageNode（递归核心结构）

```jsonc
{
  "type": "text",          // 类型枚举见 3.1
  "text": "消息文本",       // 所有类型必有：文本原文或占位描述（"[图片]" 等）

  // —— 按 type 附加，无关字段不出现 ——
  "media": {               // image | voice | video | file | emoji
    "url": "https://hub.example.com/files/xxx",   // 平台代理 URL；failed 时为 null
    "status": "ready",     // ready | failed（下载重试耗尽，降级投递）
    "thumbnailUrl": "https://.../thumb",
    "fileName": "报表.xlsx",
    "mimeType": "application/vnd.ms-excel",
    "size": 102400,        // 字节
    "durationMs": 3000,    // 语音/视频
    "width": 800, "height": 600,                  // 图片/视频/emoji
    "md5": "..."
  },
  "link":        { "title": "...", "desc": "...", "url": "...", "thumbnailUrl": "..." },
  "miniProgram": { "appId": "...", "title": "...", "pagePath": "...", "coverUrl": "...", "sourceName": "..." },
  "location":    { "label": "...", "address": "...", "lat": 31.23, "lng": 121.47 },
  "card":        { "wxid": "...", "nickName": "...", "avatarUrl": "..." },
  "transfer":    { "amount": "10.00", "memo": "...", "direction": "receive" },
  "redPacket":   { "greeting": "恭喜发财" },

  // —— 递归字段 ——
  "items": [ /* MessageNode[]，仅 chat_record：合并转发内条目，可嵌套 chat_record */ ],
  "quote": { /* MessageNode，嵌套引用（合并转发条目内的引用等） */ },

  // —— 条目级元数据（items 内使用）——
  "senderName": "陈可乐",          // 合并转发条目只有昵称（原始数据仅 hash，无 wxid）
  "sourceMessageId": "msg_xxx",    // 源自 fromnewmsgid，可回链已入库消息
  "sentAt": "2026-07-01T12:00:00+08:00",

  // —— 兜底 ——
  "rawType": "FINDER_FEED"         // unsupported/system 时保留原始类型名
}
```

### 3.1 类型枚举

| type | 说明 | 附加字段 |
|------|------|---------|
| `text` | 文本 | — |
| `image` | 图片 | media |
| `voice` | 语音（Silk→MP3 转码后投递） | media |
| `video` | 视频 | media |
| `file` | 文件 | media |
| `emoji` | 动画表情 | media（mime=image/gif） |
| `link` | 链接分享（必须解析 XML 的 title/des/url，不许烂尾） | link |
| `mini_program` | 小程序 | miniProgram |
| `chat_record` | 合并转发/聊天记录 | items[] |
| `location` | 位置 | location |
| `card` | 名片 | card |
| `transfer` | 转账 | transfer |
| `red_packet` | 红包 | redPacket |
| `system` | 群系统事件（改群名/入群/踢人等，文本描述） | rawType |
| `unsupported` | 无法识别 | rawType |

已知类型给独立 type + 基础字段；未知一律 `unsupported` + `rawType` + 默认描述文本，后续新类型按此模式扩展。

### 3.2 引用（quote）规则

引用**不是消息类型**，是顶层/条目的可选 `quote` 字段，值为 MessageNode：

1. **本地回查优先**：`refermsg.svrid` = 被引用消息的 `newMsgId`，先查本地已入库消息，命中则从标准消息复制快照（最可靠，实测验证）。
2. 回查失败 → 解析 `refermsg.content`（HTML 反转义后二次 XML 解析，可能含第三层内嵌 XML）。
3. 仍失败 → `{ "type": "unsupported", "text": "引用了一条消息，暂未解析内容" }`。
4. quote 快照为轻量结构：不递归携带大载荷（引用的合并转发其 recorditem 在 GeWe 侧已被清空，只能给摘要）。
5. quote 上附 `senderName` / `sourceMessageId`（源自 chatusr/svrid）。

### 3.3 合并转发（chat_record）规则

- `recorditem` CDATA 内嵌 XML → `items[]`，每条是同构 MessageNode。
- `dataitem.datatype` 映射：1=text，2=image，4=video，8=file，37=emoji；`refermsgitem` → 条目的 `quote`。
- 条目发送者仅有昵称（原始数据只有 sha256 hash），`senderName` 尽力提供，无 wxid。
- 嵌套合并转发（datatype=17）天然支持：items 里再套 chat_record 节点。
- 条目媒体同样下载转平台 URL；条目级下载失败不阻塞整条消息，条目标记 `media.status: "failed"`。
- 解析失败降级：`{ "type": "chat_record", "text": "标题", "items": [] }` + 描述。

## 四、GeWe 类型映射表

回调为 v2 格式（扁平小写字段，`msgType` 字符串枚举）。

### 4.1 消息类映射

| GeWe msgType | appmsg type | → 标准 type | 备注 |
|---|---|---|---|
| TEXT | — | text | |
| IMAGE | — | image | XML: msg/img，aeskey/cdn 字段 |
| VOICE | — | voice | Silk 下载后转 MP3 |
| VIDEO | — | video | 注意 media_expire_at ≈14 天过期，及时下载 |
| EMOJI | — | emoji | cdnurl 可直接下载 |
| FILE | 6 | file | 文件"发送完成" |

### 4.2 事件类（不生成标准消息，走账号/事件处理）

| GeWe msgType | 处理 |
|---|---|
| REVOKE_MSG | 撤回流程（见第五节） |
| MOD_CONTACTS / DEL_CONTACTS / FRIEND_VERIFY / FRIEND_CONFIRM | 联系人同步流程 |
| LOGIN_SUCCESS / LOGOUT / LOGIN_ERROR / Offline / RECONNECT_* / LONG_* | 账号在线状态更新 |
| UNKNOWN / 无 msgType 报文（订阅握手、探针） | 入库 raw 后跳过，webhook 返回 200 |

### 4.3 解析陷阱清单（实现必须处理）

1. **三层解析**：JSON → content XML → 转义/CDATA 内嵌 XML（引用、合并转发）。
2. **时间单位混用**：顶层 `createTime` 毫秒；XML 内 `createtime`/`srcMsgCreateTime` 秒。统一转 ISO8601。
3. **newMsgId int64 精度**：全链路字符串。
4. **裸 `&` 破坏 XML**：recorditem 里 CDN URL 常含未转义 `&`，解析失败需正则转义重试。
5. **wxid 格式**：`fromUser` 可能是 `@openim`（企微互通），校验不能只认 `wxid_` 前缀。
6. **群聊判定**：`fromGroup` 存在且以 `@chatroom` 结尾（v2 真实发送者直接在 `fromUser`，无前缀剥离问题）。
7. **v2 无 MsgSource**：atuserlist 不可得，mentions 按 3.1 节降级策略。
8. **伪 id**：SYSTEM/functionmsg 的 msgId==newMsgId==秒级时间戳，不能当真实去重键；无 newMsgId 时降级为 `account + sha1(稳定JSON)`。
9. **实体编码**：CDATA 内 `&#x20;`/`&#x0A;`/`&quot;` 需解码。
10. **PAT_MSG 分类不可信**（网关 bug，实测样本是 dynacfg 配置下发），按 content 实际结构判定。

## 五、消息生命周期与状态

- 标准消息带 `status`：`normal` | `revoked`。
- **撤回**：收到 REVOKE_MSG → 解析 sysmsg 定位被撤回 `newMsgId` → 原消息标记 `revoked` + `revokedAt`（内容完整保留）→ 向已投递过该消息的应用推送 `message.revoked` 事件（含 messageId）；UI 显示"[已撤回]"徽标，右键可查原文。
- 撤回消息若仍在下载/处理管道中未投递，则直接以 revoked 状态入库，不再投递 created。
- 每条消息记录：来源（callback/平台发送）、去重键（`appid + newMsgId`）、投递记录（投给哪个应用、状态、时间）。

## 六、媒体处理

1. **下载完成才投递**：媒体消息在标准化阶段完成下载 + 落盘 + 生成平台代理 URL，之后才进入投递队列。不做"先投后补"。
2. **下载失败降级投递，不丢消息**：下载失败先有限重试（默认 3 次退避）；重试耗尽后**消息照常入库投递**，`media.status: "failed"`、`media.url: null`，`text`/`renderedText` 给降级描述（如 `[文件: 报表.xlsx] 下载失败`、`[图片] 下载失败`）——下游必须知道消息存在，只是内容不可得。禁止因媒体失败静默吞消息（会造成下游上下文空洞 + 会话队列被堵死）。
3. 合并转发条目/引用内媒体下载失败 → 同一套降级规则：条目标 `media.status: "failed"`，消息照常投递。
4. 失败的媒体支持后台人工重试下载，成功后更新 `media.status: "ready"` 与 URL（第一版不推送更新事件，UI 刷新可见）。
5. 语音 Silk → MP3 转码后投递（下游只见 audio/mpeg）。
6. GeWe 下载调用串行 + 间隔（官方要求 3~10 秒），同一消息只下载一次；视频/文件 CDN 约 14 天过期，入库即下。
7. 平台代理 URL 带签名/过期机制（不裸奔，GeWeCenter 的 MVP 妥协不继承）。

## 七、投递模型

### 7.1 顺序保证（A 方案）

- **per-conversation 有序，跨会话并行**：同一会话内，上一条消息未处理完（下载 + 标准化 + 推送动作完成）不投递下一条；不同会话互不阻塞。
- "处理完"= 推送动作完成，**不等下游 ACK**（ACK 用于游标推进与补发，不阻塞后续投递）。
- 可靠性：SSE 携带事件 ID，支持 `Last-Event-ID` 断点续传 + 显式 ACK 接口 + 失败重试/补偿。

### 7.2 逐条投递，不做批量事件

- 一条 SSE 事件 = 一条标准消息（`message.created`）。**不设计 `message.batch`**。
- 依据（Hermes 消费模型调研）：Hermes 入口 `MessageEvent` 是单消息结构，无数组字段；批量事件会破坏 ACK/重放/去重粒度（半批失败无法表达）；现有 gewecenter-platform 插件全部可靠性机制以单事件为单位。

### 7.3 防抖分工：配置在 Hub，执行在插件

会话级消息防抖（用户连发多条攒一批再触发 AI）的需求成立，但执行点在下游插件：

- **GeWeHub 负责配置管理**：每个会话可配 `debounceMs`（静默窗口，滚动重置）与 `maxWaitMs`（从首条起最大等待），应用提供默认值，会话可覆盖；后台 UI 在会话绑定面板配置。
- **配置随消息 metadata 下发**（并提供配置查询接口），Hub 本身始终实时逐条投递，不持有延迟状态。
- **Hermes 官方插件执行防抖**：复刻 gewecenter-platform 已验证的滚动窗口合并机制（仅纯文本攒批、非文本/引用事件强制冲刷保序、合并后统一 ACK、metadata 保留全部 message_ids）。
- 依据：Hermes 核心无用户级防抖（空闲首条毫秒级触发，默认 interrupt 模式下中台攒批快发会造成连环打断）；Hermes 团队自身将用户级防抖定位为平台适配器职责（weixin 适配器 3s/5s 窗口同模式）。
- 其他简单下游（webhook 等）不受影响，各自决定是否防抖。

### 7.4 下游事件类型

| eventType | 载荷 | 说明 |
|---|---|---|
| `message.created` | 完整标准消息 | 新消息 |
| `message.revoked` | messageId + revokedAt | 已投递消息被撤回 |

后续按需扩展（如交互事件），均单事件单载荷。

### 7.5 防自循环

- `isSelf=true`（托管微信自发/手机端发送）默认只入库不投递（可配置开启）。
- 通过 GeWeHub 发送接口产生的消息不回投给发送方应用。
