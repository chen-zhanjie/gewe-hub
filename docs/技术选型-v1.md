# GeWeHub 技术选型 v1

> 状态：已确认（2026-07-06）
> 依据：`docs/产品形态-v1.md`、`docs/标准消息结构-v1.md` 以及本轮 server / web / 插件选型讨论。
> 本文档是第一版 scaffold 和实现的技术边界依据。后续若出现性能、部署或团队维护约束变化，再通过新版本文档修订。

## 一、选型目标

GeWeHub 第一版目标不是搭一个通用后台，而是跑通稳定的 GeWe 消息中台闭环：

```text
GeWe webhook
  -> 原始回调入库
  -> 去重与标准化
  -> 媒体下载与代理
  -> 会话和应用路由
  -> SSE/ACK 投递给插件
  -> 下游通过标准发送接口回复
  -> 工作台可查看、调试、补偿
```

技术选型优先满足：

- 数据可审计：消息、投递、发送、失败原因都能在主库追溯。
- 链路可补偿：任务失败后可以重试、人工排查、重新投递。
- 第一版低复杂度：不提前引入重型队列、多租户、多通道抽象或复杂权限系统。
- 前后端类型清晰：接口契约稳定，便于 web 和插件消费。
- 插件边界清楚：GeWeHub 核心不绑定 Hermes，Hermes 通过官方插件接入。

## 二、总体技术栈

```text
server:  Node.js LTS + TypeScript + NestJS + Fastify adapter
db:      MySQL 8
cache:   Redis
orm:     Prisma
jobs:    MySQL durable outbox + NestJS worker + Redis lease/lock
api:     OpenAPI + DTO/Zod 校验

web:     React + TypeScript + Vite
ui:      Tailwind CSS + shadcn/ui + lucide-react
state:   TanStack Router + TanStack Query + TanStack Table + TanStack Virtual
forms:   React Hook Form + Zod

plugin:  Python Hermes Agent 插件
deploy:  Docker Compose
```

## 三、Server 选型

### 3.1 已确认

- 运行时：Node.js LTS。
- 语言：TypeScript。
- 框架：NestJS。
- HTTP 引擎：Fastify adapter。
- 数据库：MySQL 8。
- 缓存与轻状态：Redis。
- ORM：Prisma。
- API 契约：OpenAPI。
- 后台任务：MySQL durable outbox + NestJS worker + Redis lease/lock。

### 3.2 为什么选 NestJS

GeWeHub server 的复杂度主要来自业务链路，而不是单个 HTTP 接口：

- webhook 快速接收和异步处理。
- 消息去重、标准化、媒体下载、撤回修订。
- 会话、联系人、群、群成员软状态维护。
- 应用路由、SSE 长连接、ACK、补偿重试。
- 发送接口、发送记录、运行观测。

NestJS 的模块边界、依赖注入、测试组织和 OpenAPI 支持更适合这类长期维护的中台。底层使用 Fastify adapter，保留较好的 HTTP 性能。

### 3.3 为什么不用 Fastify/Hono 直写

Fastify/Hono 直写更轻，但第一版不是轻 API 服务。若直接按 route + handler 快速堆代码，后续消息标准化、投递状态机、发送状态机、诊断页面会很快变成隐式耦合。

因此第一版选择 NestJS 做主干，Fastify 只作为 HTTP 引擎。

### 3.4 MySQL 使用原则

- 微信 ID、GeWe appId、newMsgId、messageId 一律按字符串保存和传输，避免 int64 精度问题。
- raw payload、normalized message 可使用 MySQL JSON 字段保存。
- 关键查询字段必须冗余为普通列，例如 `account_id`、`conversation_id`、`message_id`、`event_type`、`status`、`sent_at`。
- 所有可审计状态都以 MySQL 为事实源，Redis 不作为唯一可靠存储。
- 同步类数据采用软状态模型：upsert + 状态标记，不物理删除。

### 3.5 Redis 使用原则

Redis 只承担轻状态和并发辅助：

- SSE 连接状态。
- 短期锁和 worker lease。
- per-conversation 投递锁。
- 临时 dedupe cache。
- 限流和短期运行状态。

Redis 不承担唯一可靠队列，不保存必须长期审计的数据。

### 3.6 后台任务模型

第一版不引入 BullMQ。采用 MySQL durable outbox：

- webhook 入站后先写 `webhook_events`，快速返回 200。
- worker 轮询或按节奏扫描 pending 任务。
- 通过 lease 字段或 `SELECT ... FOR UPDATE SKIP LOCKED` 抢占任务。
- 处理完成后更新状态、失败原因、重试次数、下次重试时间。
- 投递、发送、媒体下载等任务都在 MySQL 中可查询、可补偿。

建议任务表按职责拆分或使用统一 outbox 表加 task type：

- webhook 处理任务。
- 消息标准化任务。
- 媒体下载任务。
- 下游投递任务。
- 发送任务。

后续如果吞吐量或任务调度复杂度明显上升，再评估引入 BullMQ 或其他队列系统。

## 四、Web 选型

### 4.1 已确认

- 框架：React + TypeScript。
- 构建工具：Vite。
- UI 基础：Tailwind CSS + shadcn/ui。
- 图标：lucide-react。
- 路由：TanStack Router。
- 服务端状态：TanStack Query。
- 表格：TanStack Table。
- 长列表和聊天虚拟滚动：TanStack Virtual。
- 表单：React Hook Form + Zod。

### 4.2 shadcn/ui 的定位

shadcn/ui 只作为通用组件体系，不作为完整后台框架，也不作为聊天 UI 框架。

适合使用 shadcn/ui 的部分：

- 按钮、输入框、下拉、弹窗、抽屉、Tabs。
- 状态 Badge、复制按钮、确认弹窗。
- 右键菜单、命令面板、Toast。
- 设置页、表单页、日志页基础组件。

需要自研的部分：

- 仿微信聊天工作台布局。
- 会话列表和消息气泡。
- 不同消息类型渲染。
- 消息右键详情。
- 原始 payload / 标准化 JSON / 投递记录调试面板。
- 聊天区虚拟滚动和定位。

### 4.3 第一版页面边界

第一版 Web 主屏是聊天工作台，不是传统 CRUD 后台。

主屏：

- 左侧：微信账号切换 + 会话列表。
- 中间：聊天区、消息搜索/筛选、文本/图片/文件发送。
- 右侧：消息详情、联系人信息、会话绑定应用、主人、主渠道、投递过滤器、防抖参数、调试面板。

独立页面：

- 登录页。
- GeWe 接入设置。
- 应用管理。
- 账号/联系人/群/群成员管理。
- 推送日志。
- 发送记录。
- 运行观测。
- 文件资源。

### 4.4 不采用的前端路线

- 不采用 Vue / Element Plus：GeWeHub 是新项目，不继承 GeWeCenter 的前端技术包袱。
- 不采用 Ant Design Pro / Arco Pro 作为主形态：CRUD 起步快，但容易把产品带回传统后台，不符合“仿微信聊天工作台优先”。
- 不只用 headless 组件：自由度高，但第一版样式和组合成本偏高。

## 五、Plugin 选型

### 5.1 首期目标

首期只维护 `plugins/hermes-agent/` 官方插件。其他下游平台先保留目录约定，不提前实现。

### 5.2 技术选型

- 语言：Python。
- 宿主：Hermes Agent 插件体系。
- 输入：GeWeHub 标准消息信封。
- 输出：Hermes 可理解的消息事件。
- 发送：通过 GeWeHub 标准发送接口回复。

插件不因为 GeWeHub server 使用 TypeScript 而改成 TypeScript。插件应贴近宿主生态，降低安装和运行成本。

### 5.3 插件职责

插件可以做：

- 连接 GeWeHub SSE。
- 持久化 `Last-Event-ID`。
- 调用 ACK 接口。
- 短期去重。
- per-conversation 顺序处理。
- 执行会话级防抖。
- 将标准消息转换为 Hermes `MessageEvent`。
- 调用 GeWeHub 标准发送接口。
- 提供配置、诊断和错误提示。

插件不做：

- 直接调用 GeWe 平台。
- 解析 GeWe raw payload。
- 自己维护消息主存储。
- 绕过 GeWeHub 的发送、审计、路由和权限边界。
- 实现 AI 能力本身，例如知识库检索、自动回复策略、persona 和工具调用。

### 5.4 插件配置

建议配置项：

- `GEWEHUB_BASE_URL`
- `GEWEHUB_APP_ID`
- `GEWEHUB_APP_TOKEN`
- `GEWEHUB_SSE_URL`
- `GEWEHUB_ACK_ENABLED`
- `GEWEHUB_DEBOUNCE_MS`
- `GEWEHUB_MAX_WAIT_MS`
- `GEWEHUB_HOME_CONVERSATION_ID`
- `GEWEHUB_LOG_LEVEL`

Hub 后台负责生成 SSE URL、token 和复制配置片段；插件负责在 Hermes 侧读取并运行。

## 六、共享契约

建议新增 `packages/contracts`，作为 server、web、plugin 的契约来源。

内容包括：

- 标准消息 schema。
- `message.created` / `message.revoked` 事件定义。
- 发送请求和响应定义。
- ACK 请求定义。
- 应用配置和会话绑定配置定义。
- 错误码和错误响应格式。

输出形式：

- TypeScript types：供 server 和 web 使用。
- JSON Schema：供接口校验和文档生成使用。
- Python models：供 Hermes 插件使用，可由 JSON Schema 或手写 Pydantic models 维护。

原则：

- 标准消息结构以 `docs/标准消息结构-v1.md` 为产品依据。
- 下游插件只接收标准层，不接收 GeWe raw payload。
- schemaVersion 必须显式保留，插件遇到不支持版本时应给出明确错误。

## 七、部署选型

第一版使用 Docker Compose：

```text
server
web
mysql
redis
nginx 或 caddy
```

建议：

- 本地开发可以 server、web 直接启动，MySQL/Redis 用 compose。
- 生产部署先使用单机 compose，降低运维复杂度。
- 文件存储先支持本地目录，接口和配置预留 S3-compatible。
- 后续如需要水平扩展，再拆分 worker、SSE server、对象存储和独立反向代理策略。

## 八、明确暂不采用

第一版暂不采用：

- BullMQ：先用 MySQL durable outbox，保证可审计、可补偿、低复杂度。
- PostgreSQL：本项目主库确认使用 MySQL 8。
- PHP / MineAdmin：只参考 GeWeCenter 业务经验，不继承技术栈。
- Vue / Element Plus：不作为 GeWeHub 新前端路线。
- 多 GeWe key 在线管理：第一版单 key env 配置。
- 多用户 / RBAC：第一版单管理员。
- 多通道微信抽象：第一版只接 GeWe API。
- mobile 端：只预留目录，不进入第一版主线。

## 九、后续演进触发条件

只有在出现明确需求或瓶颈时才升级复杂度：

- 任务吞吐或调度复杂度明显上升：评估 BullMQ、独立 worker 集群或消息队列。
- 文件量和带宽变大：启用 S3-compatible 对象存储和 CDN。
- 多实例 SSE 复杂度上升：拆分 SSE gateway，强化 Redis 连接状态协调。
- 多团队或多客户使用：评估多管理员、RBAC、审计权限和多租户边界。
- 多下游平台需求稳定：在 `plugins/` 下新增独立插件包，不污染 server 核心。

## 十、当前定论

第一版技术栈正式确定为：

```text
NestJS + Fastify + MySQL + Redis + Prisma
MySQL durable outbox + NestJS worker
React + Vite + TanStack + Tailwind + shadcn/ui
Python Hermes Agent plugin
Docker Compose
```

后续 scaffold 和实现应以本文档为准。
