# GeWeHub 产品形态 v1

> 状态：已定稿（2026-07-05 产品讨论 + 后续两轮确认）
> 本文档是第一版开发的产品边界依据。与 AGENT.md 冲突时，以本文档为准。

## 一、产品定位

GeWeHub（个微中枢）是**仿微信聊天工作台优先**的 GeWe 消息中台，三层结构：

```
微信管理层  →  消息中台层  →  应用路由层
```

- **微信管理层**：管理 GeWe 接入 key、微信账号、登录状态、联系人、群、群成员、头像、备注、标签。
- **消息中台层**：接收 webhook、保存原始 payload、去重、标准化、维护会话、聊天记录查询、文件存储、下游推送、统一发送、消息调试。
- **应用路由层**：Hub 应用跨微信绑定会话，配置主人、主渠道、微信账号备注，通过插件（默认 Hermes Agent）把标准消息投递给下游 AI。

AI（Hermes / Codex / Dify 等）只通过插件消费 GeWeHub 标准消息，**不直接接触 GeWe**。

## 二、核心概念模型

### 2.1 GeWe 侧概念只是接入凭证，不进业务层

GeWe 平台侧存在「应用（每应用一个 key/Token）→ 微信节点（appId/wxid）」两级概念。GeWeHub **不把 GeWe 应用作为业务概念**，它只是接入凭证：

```
接入层（内部）：GeWe 应用 key（第一版单 key，env 配置）
   职责：接收回调、调用 GeWe API、维护 wxid ↔ appId 映射
        ↓ 消息按 wxid 归属
业务层（核心）：微信账号（wxid 唯一）
   └─ 会话 =（微信账号, 对端 id）    ← 平台最小业务单元
        └─ 消息
消费层：Hub 应用 ──绑定──> 会话（可跨微信账号绑定多个会话）
```

- 平台业务层**只有微信账号，没有 GeWe 应用**；wxid 是账号身份，appId 只是发送/调 API 时需要的设备参数。
- wxid ↔ appId 映射为纯内部数据，仅在「接入设置」页可见。

### 2.2 GeWe 接入事实（调研结论，来自官方文档）

- 一个 GeWe Token（key）可绑定多个微信；每个微信登录后对应一个 appId（设备 ID）。
- 回调配置是 **Token 级别**：一个 Token 配一个回调地址，该 Token 下所有已登录微信的消息都推到该地址。
- 每条回调自带 `appid` + `wxid`；官方去重 key 为 `appid + newMsgId`。
- 回调地址除在 GeWe 后台配置外，也可通过 `POST /gewe/v2/api/login/setCallback` 接口设置。
- **没有**「列出 Token 下所有 appId」的接口，appId 只能来自登录流程返回或回调消息自带。
- 回调接口必须公网可访问，3 秒内响应；接口发送的消息不触发回调，手机端发送的会触发（多端同步）。

## 三、第一版核心规则（铁律）

1. **单 GeWe key，env 配置**
   - 第一版只支持一个 GeWe 应用 key，写在 env（如 `GEWE_TOKEN`、`GEWE_BASE_URL`），不做在线更换和多 key 管理。
   - 回调 URL 固定为 `/webhook/gewe/{secret}`（secret 放 env 或首次启动生成），用于防伪造——GeWe 回调无签名机制，裸 URL 任何人可伪造消息。
   - 后台「接入设置」页展示回调 URL，提供「一键设置回调」（调 setCallback）和「复制 URL 手动配置」两种关联方式。
   - 启动时自动验证 key 有效性并展示状态。
   - 后续如需多 GeWe 应用，再扩展为 DB 管理 + per-key 回调 URL；数据模型不为此提前复杂化。

2. **微信账号：自动发现 + 手动录入，均可编辑**
   - 自动发现：回调出现新 `appid`/`wxid` 时自动注册账号记录，再调 API 补全昵称、头像、在线状态。
   - 手动录入：支持手动添加账号（填 appId/wxid）。
   - 所有账号记录支持编辑（备注、appId 修正等）。

3. **会话是应用归属的最小单元，一个会话最多绑定一个应用**
   - 会话默认未绑定；未绑定会话所有消息只入库，不投递。
   - 会话绑定应用后，**默认全量投递**该会话的所有消息（私聊和群聊一致）。
   - "是否处理"是下游的语义决策：标准消息携带 `is_at_me`、`mentions[]` 等 @ 元数据，下游（如 Hermes 的"群聊仅 @ 才处理"）自行判断。中台提供判断依据，不代做判断。
   - 会话绑定配置提供可选投递过滤器：`全部消息（默认）/ 仅 @ 机器人`，为不具备过滤能力的简单下游兜底。
   - 换应用必须显式解绑/转移绑定。
   - 每条消息记录投递时命中的应用；会话改绑后历史消息归属可追溯。
   - 第一版不做多应用路由、路由优先级、multi-cast、冲突检测。

4. **单主人**
   - 每个应用配置一个主人联系人 + 主渠道（消息通知推送到主渠道）。
   - 多主人（不同主人不同应用）第一版不做。

5. **群聊上下文**
   - 默认全量投递下，绑定后的群聊消息会持续推给下游，下游天然持有实时上下文。
   - 提供聊天记录查询接口：按会话 ID 或消息 ID 向前/向后翻页查 N 条，供下游拉取绑定之前的历史或补充上下文。
   - 可选便利项：投递事件可携带最近 N 条历史（N 应用可配），供刚接入/重连的下游快速恢复上下文。

6. **单管理员**
   - 后台仅一个管理员，账号与密码 hash（bcrypt/argon2）放 env，提供 hash 生成脚本。
   - 不做注册、找回密码、多用户、角色、RBAC。

7. **只接 GeWe API**
   - wxauto 系列仅作调研参考；第一版不做通道抽象、不接其他微信通道。

8. **推送选型：SSE 优先，逐条投递，会话内有序**
   - SSE 长连接推送，带 `Last-Event-ID` 断点续传、显式 ACK、状态记录、失败原因、重试/补偿。
   - **逐条投递**：一条 SSE 事件 = 一条标准消息，不做批量事件（Hermes 入口为单消息模型，批量会破坏 ACK/重放/去重粒度）。
   - **顺序保证**：per-conversation 有序——同一会话内上一条未处理完（媒体下载 + 标准化 + 推送完成）不投递下一条；跨会话并行互不阻塞；不等下游 ACK 阻塞后续投递。
   - **媒体下载完才投递**：不做"先投后补"；下载失败有限重试后**降级投递**（`media.status:"failed"` + 描述文本如"[文件] 下载失败"），不丢消息、不堵塞会话队列。
   - 预留 Webhook 推送。
   - 应用接入方式：复制一条 URL + token 即可配置进 Hermes 插件。

9. **会话级消息防抖：配置在 Hub，执行在插件**
   - 需求：用户连发多条消息时攒一批再触发 AI，避免逐条抢答。
   - 每个会话可配 `debounceMs`（静默窗口，滚动重置）与 `maxWaitMs`（首条起最大等待）；应用提供默认值，会话可覆盖；后台在会话绑定面板配置。
   - GeWeHub 始终实时逐条投递，不持有延迟状态；防抖配置随消息 metadata 下发（另提供查询接口）。
   - Hermes 官方插件执行防抖合并（复刻 gewecenter-platform 已验证的滚动窗口机制：仅纯文本攒批、非文本事件强制冲刷保序、合并后统一 ACK）。
   - 依据：Hermes 核心无用户级防抖，且默认 interrupt 模式下中台攒批快发会造成连环打断；用户级防抖属于平台适配器职责。

## 四、应用与备注体系

### 4.1 应用可跨微信绑定会话

Hub 应用与微信账号是**多对多**关系：

- 一个应用可以绑定来自不同微信账号的会话（如应用 A 绑定 wxid₁ 的 3 个会话 + wxid₂ 的 2 个会话），统一作为一个应用处理。
- 同一个微信的不同会话也可以分别绑定到不同应用。
- 约束不变：一个会话最多绑定一个应用。

因此投递给下游的标准消息必须携带**接收微信上下文**：是哪个微信收到的消息（wxid、昵称、平台备注、该应用对它的备注）。

### 4.2 备注模型

由于会话与应用是 1:1 绑定，会话及其成员永远只有一个应用视角，**不需要应用级的会话/成员备注**；只有微信账号与应用是多对多，才需要应用视角备注：

| 对象 | 平台级备注 | 应用级备注 | 原因 |
|------|:---:|:---:|------|
| 微信账号 | ✅ | ✅ | 账号:应用多对多，不同应用视角可不同 |
| 会话（私聊/群） | ✅ | ❌ | 会话:应用 1:1，无视角冲突 |
| 会话内成员（含群成员） | ✅ | ❌ | 备注跟随会话，同样 1:1 |

- 平台级备注由管理员在工作台维护，全局可见。
- 微信账号的应用级备注由应用维护，投递时优先取应用级、回落平台级。

**投递时组装**：标准消息注入接收微信（wxid + 备注）、会话备注、发送人/被@人的成员备注。下游 AI 无需二次查询。

备注编辑入口在会话/联系人页面点选完成，**不要求手动输入微信 ID**（联系人信息由 API 拉取）。

## 五、数据原则（全局硬约束）

GeWeHub 是事实的记录者，不是微信状态的镜像。微信里"消失"的东西（退群、删好友、撤回）在平台里是**发生过的事实 + 一个新状态**，不是数据消失。这是审计、历史追溯、AI 上下文和调试能力的基础。

### 5.1 软状态模型：只增改状态，禁止物理删除

联系人、群、群成员、会话等同步类数据，同步时 **diff 后只做 upsert + 状态标记，绝不 delete**：

| 对象 | 状态（示意） |
|------|------|
| 联系人 | `active` / `deleted`（被删/删人）/ `blocked` |
| 群成员 | `active` / `left`（退群）/ `removed`（被踢） |
| 群 | `active` / `disbanded`（解散）/ `quit`（自己退群） |
| 会话 | `active` / `archived` / `inactive` |

- 每次同步记录 `last_synced_at`；本次同步未出现的成员标记为 `left`/`removed`，不删除。
- 状态变更记录时间戳（如 `left_at`），条件允许时保留状态变更历史。
- 状态推导两个来源：**回调事件**（实时）+ **定时全量同步 diff**（兜底，GeWe 不一定推送所有变更事件）。
- 群成员备注因此天生安全：人退群备注仍在，重新进群可续用；历史消息发送人即使已退群也能正确关联。
- UI：非 active 对象灰显 + 状态徽标，不消失；下拉选择器（如群成员选择）默认只展示 active，可切换查看全部。
- 平台内的选择类交互（选联系人、选群成员等）数据源一律来自平台本地数据，不实时查询 GeWe API。

### 5.2 消息可修订：撤回是状态，不是删除

微信撤回通过回调告知（系统消息指向被撤回消息 id）：

- 收到撤回事件 → 定位原消息 → 标记 `is_revoked = true` + `revoked_at`，原消息内容和标准结构**完整保留**。
- 撤回事件本身也作为一条系统消息入库（谁撤回了哪条）。
- UI：原消息位置显示"[已撤回]"徽标，右键仍可查看原始内容、payload 和撤回事件。
- 下游投递：撤回作为消息状态更新事件推送（如 `message.revoked`），让下游知道"那条消息被撤回了"，而非假装未发生。
- 标准消息模型必须预留 `status` 字段（normal/revoked/…），消息生命周期支持被后续状态事件修订。

## 六、标准消息结构原则

> 完整 schema、类型枚举、GeWe 映射表与解析陷阱清单见 `docs/标准消息结构-v1.md`。

1. **递归 MessageNode**：引用、合并转发、多层嵌套转发都复用同一节点结构（content/quote/items 递归），不为复杂消息散落特例类型。AI 通过稳定嵌套格式处理。
2. **已知类型独立 type + 基础字段**（text/image/voice/video/file/emoji/link/mini_program/chat_record/location/card/transfer/red_packet/system）；不可识别归 `unsupported` + `rawType` + 默认描述，永不 null 炸弹。
3. **三层结构**：`raw`（原始 payload，仅平台内部，用于追溯）/ `normalized`（标准层，系统与下游处理）/ `rendered`（统一展示摘要 `renderedText`）。
4. **@ 元数据**：群聊消息携带 `isAtMe`（源自 pushContent，可靠）、`mentions[]`（昵称尽力匹配群成员，失败标 `resolved:false`——GeWe v2 回调无 atuserlist，协议上限）。
5. **接收微信上下文**：消息携带接收微信信息（wxid、昵称、平台备注、该应用对它的备注），下游可区分消息来自哪个微信。
6. **消息状态**：标准消息带 `status`（normal/revoked）；撤回以 `message.revoked` 独立事件投递。
7. **去重**：`appid + newMsgId`（字符串处理防 int64 精度丢失）；无 newMsgId 降级稳定哈希。
8. 下游只接收标准消息，不依赖 GeWe 原始字段；媒体一律平台代理 URL（带签名），CDN 参数不出平台。

## 七、主界面信息架构（仿微信聊天工作台优先）

第一版主屏是仿微信聊天工作台，不是传统 CRUD 后台：

- **左侧**：微信账号切换 + 会话列表（显示绑定应用徽标、未读/最近消息）。
- **中间**：仿微信聊天区，支持查看消息、搜索、筛选、发送文本/图片/文件；展示参考微信桌面端架构（小程序等消息类型友好展示）。
- **右侧**：消息详情 / 联系人信息（含平台备注标签）/ 当前会话绑定的应用、主人、主渠道、投递过滤器（全部 / 仅@）、防抖参数（debounceMs/maxWaitMs）/ 调试面板。
- **消息右键**：所有消息支持右键查看详情、原始 payload、标准化 JSON、路由/投递记录、发送/投递状态。
- **独立页面**：GeWe 接入设置（回调 URL、一键设置回调、key 状态）、应用管理（含它绑定了哪些会话）、推送日志、发送记录、运行观测、文件资源。

## 八、账号维度

- 数据模型天然多账号：所有会话/消息/联系人数据带账号维度（wxid/account_id）。
- UI 第一版默认单账号视图，支持账号切换（下拉/侧栏），不额外做复杂多账号管理页。

## 九、第一版不做（非目标）

- 多主人配置与复杂多路由。
- 多 GeWe 应用 key 管理（单 key env 配置，后续按需扩展）。
- 中台代替下游做"是否处理消息"的语义判断（仅提供 @ 元数据和可选投递过滤器）。
- 多用户后台、RBAC、注册/找回密码。
- wxauto 或其他微信通道、通道抽象层。
- 完整 CRM、工单、客服排班。
- mobile 端（仅预留目录）。
- 把 GeWe 应用概念暴露为业务概念。

## 十、两平台关联流程（用户视角）

1. env 配置 GeWe key，启动平台（自动验证 key 有效性）。
2. 后台「接入设置」点「一键设置回调」，或复制回调 URL 到 GeWe 后台粘贴。
3. 微信来消息 → 账号自动出现在账号列表（或手动录入）。
4. 会话出现在工作台 → 给需要 AI 的会话绑定应用。
5. 应用页复制 SSE URL + token → 填入 Hermes 插件 → 闭环完成。

## 稳定消息 ID、发送策略与执行模式

统一发送接口 `POST /api/send` 使用两个独立维度：

- `deliveryMode=immediate|discard|confirm` 表达发送策略；
- `executionMode=sync|async` 表达执行方式，默认 `sync`。

所有新入站和出站消息在创建时生成稳定 GeWeHub 消息 ID（`msg_<22 位 Base64URL>`）。发送成功只在 Message 上补充平台消息映射和实际发送时间，不替换稳定 ID。引用使用 `replyToMessageId`，撤回使用 `POST /api/messages/:messageId/revoke`，两者都只接受稳定消息 ID。

`immediate` 事务性预建 Message、SendRequest 和 outbox；新任务会主动唤醒进程内全局串行协调器，不等待 2 秒恢复轮询。`sync` 等待平台明确结果后返回；`async` 可靠入队后立即返回。标准成功响应只包含 `success`、`messageId`、可选 `url` 和异步的 `accepted:true`，不向 Agent 暴露 pending、held 或 SendRequest ID。

`discard` 和 `confirm` 当前均创建 `isSent=false` 的本地标准消息，不调用 GeWe。工作台中分别显示“未发送”和“待确认”，并使用低透明度、灰色调和虚线边框；人工点击发送后复用同一 Message 和稳定 ID，成功后按实际 `sentAt` 重排。

Hermes 等插件负责把普通文本或标准 JSON final 统一标准化为 `/api/send` 请求并透传两个模式；是否发送、暂存或等待确认由 GeWeHub 决定。插件默认同步，只解析标准响应，并仅使用上下文中的稳定消息 ID 完成引用和撤回。旧布尔 `send`、平台原始消息 ID 和 SendRequest 撤回协议已删除，不保留兼容分支。
